Phase 2 · Sketch Outbound HTTP and TLS

Expose wasi-http outbound as a host capability and align the existing TLS surface so guests have a complete client-side networking story for HTTPS, databases, and telemetry. Detailed sub-tasks are intentionally deferred: the wasi-http wiring depends on the WASI Preview 2 surface that phase 1 (sub-task 1f) actually ships, and the concrete shape of that surface determines what work this phase still needs.

This is a sketch, not a detailed plan. The detailed sub-task breakdown is written after phase 1f lands. Concrete wasi-http wiring, the choice between using the wasmtime-wasi-http crate directly versus building a custom outbound HTTP host capability, and the rustls version aligned with the modernised wasmtime stack are all open until then.
Status — implemented, in review. With phase 1f's component-model path in place, Phase 2 has landed as two stacked PRs:
  • 2a — the 14 existing tls_* host functions are now asserted in wat/all_imports.wat.
  • 2b–2e — outbound HTTP adopts wasmtime-wasi-http (decision 2b). WasmtimeRuntime::compile_component_with_http wires wasi:http/outgoing-handler onto the component linker (2c); a can_outbound_http per-process flag gates it (2d) — when off, the plain compile_component leaves the wasi:http imports unsatisfied so the component cannot instantiate (the deny path). An integration test drives a real component guest issuing a GET against a local server, plus a deny test (2e).
Carried forward: the finer-grained per-origin allow-list (part of 2d) is a follow-up — the hook exists (WasiHttpHooks::send_request) but only the coarse per-process on/off gate is wired so far. Inbound HTTP remains out of scope (framework concern). HTTPS works in production via the host TLS stack; the test uses plain HTTP to a local server to avoid trusting a throwaway cert.

Outbound HTTP Not present. No wasi-http, no bespoke HTTP client host capability.

Likely scope

Best-guess sub-tasks, to be refined after phase 1f:

  • 2a · Reconcile existing TLS surface. Add the tls_* host functions to wat/all_imports.wat; verify the assertion test passes. Pure documentation/audit work, can land independently of the rest of the phase.
  • 2b · Decide outbound HTTP architecture. Three options, decision deferred: adopt wasmtime-wasi-http wholesale; build a thin custom host capability with rustls + hyper; expose only socket-level capabilities and let guests link an HTTP client. Default lean: wasmtime-wasi-http once Preview 2 is reachable from phase 1f.
  • 2c · Implement chosen path. Wire the host capability, gate by per-process allow-list for hosts (analogous to existing fs preopen scoping), expose the imports to guests.
  • 2d · Permission model. Add config flags so a process can be denied outbound HTTP or restricted to specific origins. Mirrors the existing can_* flags pattern in DefaultProcessConfig.
  • 2e · Validation. Integration test: a guest makes an HTTPS request to a local test server and reads the response. Deny-list test: a guest denied outbound HTTP traps on attempted call. wat/all_imports.wat updated; gap table on docs/index.html updated.

Open questions to resolve after phase 1f

  • Component model vs module. wasmtime-wasi-http is component-model native. Do we expose this surface only to component-form guests, or build a module-form shim? The answer depends on what 1f wires up.
  • Streaming bodies. Large request/response bodies stream rather than buffer. Decide whether the guest API surfaces streams (matches WASI Preview 2 design) or buffers (simpler, but problematic for large bodies).
  • rustls server certs. Existing tls_accept takes a server config; the format is in the host. Confirm post-phase-1 the same path still works or needs an update.
  • HTTP server (inbound). Out of scope for this phase. Inbound HTTP belongs to a future Phoenix-style framework, not the substrate. Document this boundary.

Dependencies

  • Phase 1 must complete through at least sub-task 1f (WASI Preview 2 alongside Preview 1).
  • No dependency on phases 3, 4, or 5.

Definition of done for phase 2

A guest can make an outbound HTTPS request from a Lunatic process, gated by per-process permissions. The TLS surface that already exists in the codebase is asserted in wat/all_imports.wat. Detailed sub-task definitions of done will be added when the sketch above is upgraded to a full plan.
