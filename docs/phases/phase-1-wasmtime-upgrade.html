Phase 1 · Detailed plan Wasmtime + WASI Preview 2 upgrade

Bring the workspace from wasmtime = "8" to current (~v27 at time of writing) and adopt WASI Preview 2 alongside Preview 1 so existing guests keep working. This is a workspace-wide refactor with no behavioural changes intended — every existing test should pass on completion. New Preview 2 capabilities are added but not yet exposed to guests beyond a smoke-test surface; the real use of Preview 2 (outbound HTTP, sockets) lands in phase 2.

Goal

Modernise the wasmtime stack so subsequent phases can rely on current APIs (component model, wasi-http, pooling allocator, improved async host functions) without further version migration work. End state: the workspace builds and tests pass against a current wasmtime release; wat/all_imports.wat still asserts the full host-function surface; no guest-visible behaviour changes.

1a

Dependency bump and compile-error inventory

PR scope: dependencies + diagnostics

Bump wasmtime, wasmtime-wasi, wasi-common, and wiggle in the root Cargo.toml's [workspace.dependencies] to current. Do not yet fix the resulting compile errors — instead, capture the full cargo check --workspace output, classify the errors by category (fuel API, ResourceLimiter, WASI surface, func_wrap signatures, other), and use that classification to refine the remaining sub-task scopes.

  • Single commit bumping wasmtime / wasmtime-wasi / wasi-common / wiggle versions.
  • Run cargo check --workspace --all-features; capture output to docs/phases/phase-1-error-inventory.md (not committed; a working artifact).
  • PR body lists error categories and their counts.

Definition of done for 1a: branch builds Cargo.lock resolution cleanly even though cargo check fails; error inventory captured and summarised in the PR description. Tests and lint do not yet pass — this PR is intentionally non-green and will only merge when 1b–1g land alongside, either as a stacked series or as additional commits before this PR opens.

1b

Fuel and Store API migration

PR scope: lunatic-process runtime

Replace Store::out_of_fuel_trap() and Store::out_of_fuel_async_yield(n, k) with the current fuel API (Store::set_fuel, Store::fuel_async_yield_interval). Preserve the existing behaviour: out-of-fuel raises an async yield to allow the scheduler to preempt, then continues; configured limits remain those from DefaultProcessConfig.

  • Update crates/lunatic-process/src/runtimes/wasmtime.rs.
  • Adjust crates/lunatic-process/src/config.rs if any constructor signatures move.
  • Verify UNIT_OF_COMPUTE_IN_INSTRUCTIONS still maps onto the new yield interval.
  • Tests: confirm fuel-exhaustion behaviour in benches/benchmark.rs still completes; if needed, add a small integration test that runs a long-loop guest and asserts preemption.
1c

ResourceLimiter migration

PR scope: lunatic-process state

Update the ResourceLimiter impl on ProcessState to the current trait signature. Recent wasmtime versions added an async ResourceLimiterAsync variant and changed return types on some methods. Decide whether to implement the sync or async variant — sync should be enough since the existing limits are constant-time checks.

  • Update crates/lunatic-process/src/state.rs.
  • Confirm store.limiter(|state| state) wiring in runtimes/wasmtime.rs still applies.
1d

Per-crate func_wrap audit

PR scope: all lunatic-*-api crates · likely 2–3 sub-PRs

Walk each host-API crate and adjust any Linker::func_wrap / func_wrap_async calls whose signatures fail to compile under the new wasmtime version. Most should be mechanical (Send/Sync bound tightening, closure return types). Group related crates into single PRs:

  • Networking group: lunatic-networking-api, lunatic-distributed, lunatic-distributed-api.
  • Messaging & process group: lunatic-messaging-api, lunatic-process-api, lunatic-timer-api, lunatic-registry-api, lunatic-error-api, lunatic-trap-api, lunatic-version-api.
  • Storage & metrics: lunatic-sqlite-api, lunatic-metrics-api, lunatic-stdout-capture (wiggle bump).
  • Common helpers: lunatic-common-api last, after concrete crates stabilise the call sites it serves.

Each grouped PR keeps the workspace closer to green but may still fail to build until siblings land. The order is flexible; pick whichever group has the smallest diff first to validate the migration shape.

1e

WASI Preview 1 via current wasmtime-wasi

PR scope: lunatic-wasi-api

Current wasmtime-wasi splits Preview 1 into a wasmtime_wasi::preview1 module and reworks construction (WasiCtx builder shape, WasiCtxBuilder, add_to_linker signatures). Port lunatic-wasi-api to the new Preview 1 surface without changing what guests see — same environment variables, command-line args, preopen-dir host functions.

  • Update crates/lunatic-wasi-api/src/lib.rs.
  • Verify the three host functions in wat/all_imports.wat (config_add_environment_variable, config_add_command_line_argument, config_preopen_dir) retain identical signatures.
  • Run an existing WASI-using guest (e.g. examples/native_process.rs or the cargo test path) end-to-end.
1f

WASI Preview 2 alongside Preview 1

PR scope: lunatic-wasi-api + lunatic-process linker

Add WASI Preview 2 as a parallel surface, gated by a per-process config flag. Default off so existing guests continue to use Preview 1. Preview 2 requires the wasmtime component model (Component and component::Linker) rather than the classic Module + Linker path.

  • Add a can_use_wasi_preview_2 config flag (mirrors existing config_can_* permissions).
  • Introduce a parallel component::Linker wiring path in lunatic-process that activates only for components.
  • Add a smoke-test guest module compiled as a component that uses a Preview 2 capability (a simple wasi:filesystem read is enough).
  • Document in wat/all_imports.wat what is asserted vs Preview 1 only; the Preview 2 surface is not yet exposed to module-form guests.

Decision point: whether to wire wasi-http outbound here or defer to phase 2. Default position: defer; phase 1 only validates that Preview 2 can be reached and that the linker path works.

1g

Validation

PR scope: final consolidation
  • cargo build --workspace --all-features --release green.
  • cargo test --workspace green; no pre-existing tests removed.
  • cargo fmt --check and cargo clippy --workspace --all-targets green; new warnings triaged.
  • cargo build --benches green; cargo bench runs to completion (numbers are not gates, but the bench must not regress catastrophically).
  • wat/all_imports.wat still asserts every exposed host function.
  • Smoke run: an existing example (examples/native_process.rs) and the cargo test-via-WASM mode both work end-to-end against a sample guest.
  • Documentation in step: update the gap table in docs/index.html so the wasmtime row moves from "v8 (years old)" to current; mark phase 1 complete on docs/erlang-substrate.html.

Test strategy

The phase is a refactor; no new feature tests are required, but the safety net needs to be explicit. Three layers:

  • Existing tests, unchanged. The whole workspace's current test suite must pass. Any test that is removed or its assertions weakened during migration is a red flag and must be called out in the PR.
  • Import-signature assertion. The wat/all_imports.wat import signatures must still match the host linker exactly. This is the closest thing to an ABI contract and is the single most important regression gate.
  • End-to-end smoke run. A WASI-using guest, a TCP-using guest, and the cargo test-via-WASM mode all execute to completion. These exercise wasi, networking, and the test-runner mode dispatch in src/main.rs together.

One new test is justified by this phase: a Preview-2-component smoke test, added in 1f, that proves the component linker path is reachable. It does not need to do real work.

Open questions

  • Pooling allocator. Current wasmtime supports a pooling allocator that significantly speeds up activation creation. Do we adopt it in phase 1 (small additional risk), or land it as a follow-up tuning PR? Default: defer to a tuning PR after 1g.
  • Epoch interruption. Current wasmtime supports epoch-based interruption as an alternative to fuel for preemption. Fuel still works; switching is non-trivial because the scheduler is currently fuel-aware. Default: keep fuel; revisit only if fuel-based preemption regresses under the new version.
  • Component-model exposure to guests. Phase 1 wires the component linker path only for the Preview 2 smoke test. Whether the platform offers guest authors a stable component-model story is a longer conversation deferred to a future phase.
  • QUIC compatibility. lunatic-distributed uses QUIC via quinn, which depends on rustls. Not strictly part of the wasmtime upgrade but ages alongside; verify no version conflicts at 1a, escalate to a sub-task only if compile fails there.

Definition of done for phase 1

All sub-tasks 1a through 1g have merged to main; the validation checklist in 1g is satisfied; the gap-table row for wasmtime is updated; phase 2 (outbound HTTP / TLS) can begin without further wasmtime work. Per-change gates from platform definition of done apply to every sub-task PR.
