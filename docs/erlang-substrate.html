Roadmap · Substrate Completing the BEAM-style substrate

The execution model — preemptive scheduling, cheap isolated processes, mailboxes, links — is in place. The surrounding primitive surface is not. This roadmap completes the substrate so it provides the primitives an Erlang/OTP-style runtime would: monitors, node-monitor events, supervision, process groups, durable timers, persistence providers, and modern networking.

Target model

We want the host to provide the primitives, not the policy. The Erlang/OTP world spent decades settling on a small set of primitives that compose well: processes, messages, links, monitors, named registries, process groups, supervision, node membership, timers. We aim to provide those and stop there — every higher-level pattern lives in guest-side crates that authors can swap.

Phased plan

Phase 1

Wasmtime modernisation + WASI Preview 2 → detailed plan

~6–10 weeks

The single biggest unblocker. Today wasmtime, wasi-common, and wiggle are pinned at v8 — multiple years behind upstream. This blocks outbound HTTP, modern TLS, the component model, the pooling allocator, and async-host-function ergonomics. Every later phase is easier on a current runtime.

  • Bump wasmtime / wasi-common / wiggle to current.
  • Port every lunatic-*-api crate to the new wiggle / linker API.
  • Adopt WASI Preview 2 alongside Preview 1 so existing guests keep working.
  • Wire wasi-http outbound as a host capability (gated by per-process permissions).
  • Update wat/all_imports.wat to assert both the legacy and Preview 2 surfaces.

Risk Largest single chunk of work. Mitigation: do it on a long-lived branch with the existing test suite as the regression gate; do not mix with feature work.

Phase 2

Outbound HTTP & TLS for guests

~3–4 weeks · depends on phase 1

Once Preview 2 is in, expose wasi-http outbound and add tls_connect / tls_accept host functions (rustls-backed, reusing the work already done for the control-plane mTLS in 5eb9570). This unblocks storage providers, OpenTelemetry exporters, and any guest that needs to call HTTPS.

  • wasi-http outbound, with per-process allow-list for hosts.
  • TLS server + client host functions for raw TCP grains that need it.
  • Optional: blessed Rust client crates (Postgres, Redis) that use these host APIs.
Phase 3

Monitors and membership events

~1–2 weeks · independent of 1/2

Two small additive host changes that unblock most higher-level framework work. Both are Erlang-equivalent primitives; neither is currently in Lunatic.

  • Monitors with refs. monitor(pid) -> ref and demonitor(ref); delivers {DOWN, ref, reason} to the monitor's mailbox when the target exits. One-way and observation-only — unlike links, the monitor does not die with the target.
  • Node-monitor events. A subscription host function that delivers {node_up, id} / {node_down, id} to a guest process whenever cluster membership changes. Replaces today's poll-only nodes_count / get_nodes.

Phase 4

Process groups & PubSub primitive

~2–3 weeks · depends on phase 3

Erlang's pg module: named groups that any process can join, and a single host call broadcasts to all members. Used for stream fan-out, pubsub, presence, and any one-to-many delivery pattern. Implementable as an extension to lunatic-registry-api.

  • group_join(name, pid), group_leave(name, pid), group_members(name).
  • group_send(name, msg) — single host call, fan-out inside the host.
  • Cluster-aware variant that broadcasts across nodes via the distribution layer.
Phase 5

Supervision behaviour (guest-side crate)

~2 weeks · depends on phase 3

Build a blessed lunatic-supervisor crate on top of monitors + spawn. Implements the Erlang restart strategies: one_for_one, one_for_all, rest_for_one, simple_one_for_one, with intensity/period throttling and child specs. No host changes required — this is policy that belongs in user space, but it should be a blessed shared crate so frameworks don't each reinvent it.

Phase 6

Persistence provider abstraction

~3–4 weeks · depends on phase 2

A small lunatic-store-api host crate exposing a key-value-with-etag interface backed by pluggable providers. Frameworks (virtual actors, event sourcing, workflows) all need this.

  • Host trait: read(key) -> (etag, bytes), write(key, etag, bytes) -> new_etag, delete(key, etag).
  • Initial providers: SQLite (embedded), filesystem, HTTP-backed (needs phase 2).
  • ETag-based optimistic concurrency, not transactions.
  • Streaming read/write for large values.
Phase 7

Durable timers (reminders)

~2–3 weeks · depends on phase 6

Today lunatic::timer is in-memory. A reminder is a timer that survives process and silo restarts. Implementation: persist the schedule via the storage provider, on host startup replay due reminders, on fire deliver a message — activating the recipient if dormant.

  • reminder_register(target, name, due, period) -> id.
  • reminder_cancel(id).
  • Delivery: message to the named target, even if not currently running (host re-activates it).
  • Backing store: pluggable, defaults to SQLite via the persistence provider.

Out of scope for the substrate

  • Hot code loading. Replaced by rolling deploys with module/grain versioning at the framework layer.
  • ETS / Mnesia. Persistence is via storage providers; no in-runtime term store.
  • BEAM-style tracing (erlang:trace/3, observer GUI). Replaced by OpenTelemetry spans + Prometheus metrics, which is a guest-side concern once outbound HTTP works.
  • Ports / NIFs. WASM is the isolation boundary; native extensions don't fit the model.
  • Pattern matching on receive. Tag-based receive plus codegen in framework crates covers what guests need.

Definition of done

The substrate is "done" when both of the following hold:

  • Substrate-direct applications are production-ready. A team can build a supervised, distributed, persistent network service against the primitives alone — no framework required — in the Erlang/OTP tradition. This milestone lands at the end of phase 5 (supervision crate), well before framework work completes. See building on the substrate.
  • Frameworks can be implemented as ordinary crates. A third party can build a virtual-actor framework, a job queue, and a Phoenix-style web server as separate Rust crates that depend only on rust-lib, without touching any lunatic-*-api crate.

Each individual change inside any phase must also satisfy the per-change gates — tests, lint, wat/all_imports.wat updates, and documentation in step with the code. See platform definition of done for the gates that apply to every increment.
