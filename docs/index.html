Platform plan A BEAM-style substrate on WebAssembly

Lunatic is a runtime of cheap, isolated, preemptively scheduled WASM processes with mailboxes, links, and distribution. This plan completes that substrate as a programming target in its own right, and describes the first framework — Orleans-style virtual actors — that will be built on top of it as a separate Rust crate.

Build paths

Apps on the substrate

Build directly against processes, mailboxes, links, monitors, supervisors, and process groups. The patterns and roadmap milestones for this path are in Apps on the substrate.

Apps via a framework

The first framework planned is Orleans-style virtual actors, packaged as a guest-side Rust crate. Detailed phase plan in Apps via a framework.

Frameworks

Virtual actors is the framework currently scoped. The others below are placeholders — patterns the same substrate accommodates, kept here as candidates for future planning. They are not on the roadmap and have no detailed design yet.

Virtual actors

Orleans-style grains with stable identity, on-demand activation, single-activation guarantee, and persistent state.

Temporal / Cadence shape: workflows as processes whose mailbox history is the event log, activities as messages with retries.

Where the runtime is today

The current runtime supports single-node Rust services with crash-isolated workers, TCP/UDP networking, embedded SQLite, and a coordinator-style distributed mode. Several primitives that a BEAM-equivalent runtime would normally provide are absent or partial.

Definition of done

Every addition of functionality — substrate primitive, host capability, behaviour crate, framework phase — is complete only when all of the following hold. This applies per increment, not only at phase boundaries.

GateRequirement
Tests cover new code New code carries tests at the appropriate level: unit tests for libraries, integration tests for host-function changes (against a guest module exercising the call), end-to-end tests for framework features. Coverage is qualitative — every behaviour change has at least one test demonstrating it.
Full suite passes cargo test across the workspace passes, including all pre-existing tests. Per-crate tests pass for any crate touched. Bench builds are not broken (cargo build --benches).
Lint passes cargo fmt --check and cargo clippy --workspace --all-targets both pass with no new warnings.
Host-function surface is asserted Any host-function addition, removal, or rename updates wat/all_imports.wat; the import-signature test passes.
Documentation reflects reality The relevant page in /docs is updated as part of the same change — the substrate roadmap when a substrate primitive lands, the Orleans plan when a framework phase lands, the patterns guide when a new pattern becomes idiomatic, the gap table on this page when a row's status changes.
Scope rule. Phase-level "definition of done" sections on the roadmap pages describe what each phase produces. The gates above describe what every individual change within a phase must satisfy before it merges. Both apply.

Out of scope

  • Hot code loading. Replaced by module versioning and rolling deploys.
  • ETS / Mnesia equivalents. Persistence is via storage providers; no in-runtime distributed term store.
  • Ports / NIFs. WASM sandbox is the deliberate replacement for in-process native extensions.
  • Observer-style GUI and BEAM tracing. Observability is OpenTelemetry and Prometheus, not erlang:trace/3.
  • Transactions across actors / 2PC. Deferred indefinitely; sagas with compensating actions cover the same ground.
