Build path Building applications on the substrate

Applications can be written directly against the substrate's primitives: processes, mailboxes, links, monitors, supervisors, named registries, process groups, distribution, and persistence providers. This page describes the primitive surface, the canonical patterns, and the roadmap milestones that determine when this path is production-ready.

Primitive surface

The primitives a substrate-direct application uses, once the roadmap is complete:

PrimitiveWhat it provides
ProcessesCheap, isolated, preemptively scheduled units of execution with their own stack, heap, and resource budget.
MailboxesTagged message passing with selective receive; per-process FIFO.
LinksBidirectional crash coupling between two processes.
MonitorsOne-way liveness observation; receive a DOWN message when the target exits.
SupervisorsRestart strategies (one_for_one, rest_for_one, one_for_all, simple_one_for_one) with intensity / period throttling. Guest-side crate, not host primitive.
Named registryProcess names for stable addressing.
Process groupsJoinable groups for one-to-many delivery; pubsub and fan-out.
DistributionSpawn and message across nodes; node-monitor events for membership changes.
NetworkingTCP, UDP, TLS, outbound HTTP as host capabilities.
Persistence providersPluggable key-value-with-etag interface; SQLite default.
TimersVolatile (per-process) and durable (survives restart).
Resource limitsPer-process memory and fuel caps; per-process WASI permission scoping.

Canonical patterns

Supervised network service

A top-level supervisor owns an acceptor process and a worker-pool supervisor. The acceptor spawns a worker per connection under the pool supervisor. Worker crashes are contained by the pool supervisor's restart policy; the acceptor itself is supervised by the top.

Applies to: HTTP servers, custom protocol daemons, brokers, anything connection-oriented.

Stateful service with a state-owning process

A single process owns a piece of state and answers messages serially. Callers send requests with a reply-to address and correlation tag. State is loaded from the persistence provider on start and saved on change; the supervisor restarts the process on crash, which reloads from storage.

Pipeline of linked stages

Producer, transform stages, and sink, each a process, linked through the chain. Back-pressure is expressed through bounded mailboxes or explicit demand messages from downstream. A crash in any stage propagates through the links; the supervisor restarts the chain atomically.

Applies to: log processors, ETL, stream consumers, queue drains.

Sandboxed execution of untrusted or fault-prone code

Spawn the target module as a child process with strict memory and fuel caps and minimal WASI permissions. Communicate only by message. A crash, OOM, or fuel exhaustion is confined to that process.

Distributed worker pool

A named registry entry per pool; workers join a process group across nodes. Submitters send work to the group; routing is by dispatcher process or message hash. Node-monitor events from the substrate remove workers belonging to failed nodes; the pool continues with the survivors.

Substrate items required for this path

Substrate-direct applications become production-ready when the following roadmap items have landed. See Substrate roadmap for the full phase plan.

  • Monitors with refs — required for supervisors and orchestrators to detect crashes without dying with them.
  • Supervision crate — restart strategies in a blessed shared library.
  • Process groups — pubsub and fan-out as a primitive.
  • Outbound HTTP / TLS — required to call modern databases, APIs, and telemetry.
  • Persistence providers — required for state to survive restarts without bespoke I/O.
  • Node-monitor events — required for any multi-node service to react to topology changes.

The production-ready milestone lands at the end of substrate phase 5 (supervision crate). The remaining items (persistence providers, durable timers) extend what substrate-direct applications can do without changing the path.
