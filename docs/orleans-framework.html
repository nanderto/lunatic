Roadmap · Framework Orleans-style virtual actors on Lunatic

A guest-side Rust crate, lunatic-grain, that delivers Orleans semantics on top of the completed BEAM-style substrate: stable grain identity, on-demand activation, single-activation guarantee, idle deactivation, persistent state, placement strategies, durable reminders, and typed proxies. No runtime fork required — the substrate work in the sibling roadmap is sufficient.

Each silo owns a shard of the grain directory, partitioned by consistent hash on the grain id. The owner of a shard knows where every activation of grains in its range lives. On a method call, the caller's local resolver consults the local cache first, falls back to the owning silo on miss, and caches the result. The owner enforces single-activation through a compare-and-swap on the directory entry.

Call resolution sequence

sequenceDiagram autonumber participant C as Caller participant L as Local cache participant O as Directory owner participant H as Hosting silo participant G as Activation C->>L: lookup (User, 42) alt cached L-->>C: silo X else miss L->>O: lookup (User, 42) alt entry exists O-->>L: silo X (cached) else does not exist O->>O: CAS create entry O->>H: spawn activation H-->>O: pid O-->>L: silo X end end C->>G: send method message (via distribution) G-->>C: reply

Single-activation is preserved because only the directory owner can create entries, and creation goes through a CAS. Races resolve to one winner; losers redirect. On silo failure, membership events from the substrate trigger directory cleanup; the next call recreates the activation elsewhere via the same path.

Phased plan

Phase 0

Single-node scaffolding

~3–4 weeks

Prove the model end-to-end on a single node before touching distribution.

  • Grain trait with on_activate / on_deactivate hooks.
  • #[grain] proc-macro: turns an async trait into a server loop + typed proxy.
  • GrainRef<T> holds (type_id, key); GrainFactory::get<T>(key).
  • Local Directory process registered under a well-known name in lunatic::registry.
  • Idle deactivation via per-activation deadline timer.
Phase 1

Persistence provider abstraction

~3 weeks · depends on substrate phase 6

Wire grains to the substrate's storage provider. #[grain(state = MyState)] injects load on activate, save before deactivate (or on explicit save_state()), with ETag optimistic concurrency. ETag mismatch deactivates the grain so the next call reloads.

  • SQLite-backed provider (default).
  • HTTP-backed provider sketch (Postgres / Redis / S3 via outbound HTTP).
  • Optional: event-sourced variant — append events on every handler, replay on activation.
Phase 2

Distributed directory & single-activation

~6–8 weeks · depends on substrate phases 3–4

The hard part. Two design options, both proven in Orleans.

Option A: consistent-hash directory (recommended start)

Grain id hashes to a node which owns its directory entry. Membership-version-aware so ownership is stable. On membership change, the new owner handshakes with the previous owner for known entries; orphans deactivate on idle.

Option B: gossip directory

Every silo caches (grain -> silo) and gossips updates. Faster reads, fuzzier races. Orleans 7+ supports this; could be added later as an alternative directory backend.

Workstream

  • Cluster membership view from substrate node-monitor events.
  • Consistent hashing module (jump or rendezvous), keyed on node id + membership version.
  • Directory shard process per silo; CAS-based entry creation.
  • Activation race protection: only the owner can spawn.
  • Failover: on {node_down}, owner removes entries; next call reactivates.
  • Local resolver cache with retry-on-stale.
Phase 3

Placement strategies

~2 weeks · depends on phase 2

Pluggable strategy interface invoked by the directory owner when creating an activation.

  • RandomPlacement — default, picks any live silo.
  • PreferLocalPlacement — caller's silo first, fall back to random.
  • HashBasedPlacement — id hash → silo, sticky to membership version.
  • StatelessWorker — bypasses directory; multiple activations per silo for stateless work.
  • ResourceOptimizedPlacement — deferred until silo load metrics exist.
Phase 4

Reminders & observers

~2 weeks · depends on substrate phase 7
  • Reminders. Thin guest-side wrapper around the substrate's durable timer primitive. Register against a GrainRef; on fire the host reactivates the grain if dormant and delivers the message.
  • Observers. Pure guest-side. Grain holds Vec<ObserverRef>; broadcast on events. No delivery guarantee (Orleans is explicit about this); for guaranteed delivery, build streams.
Phase 5

Streams

~4–6 weeks · depends on phases 2 and 4

Two complementary implementations:

  • Stream-grain pattern. A StreamGrain per stream id uses the directory + single-activation already built. Keeps a small ring buffer, fan-outs to subscribers via process groups. Good enough for many workloads, no external dependency.
  • Broker-backed. KafkaStreamProvider / NatsStreamProvider via outbound networking. Production-grade durability and scale.
Phase 6

Interceptors, versioning, observability

~3 weeks · ongoing thereafter
  • Interceptors (IIncomingGrainCallFilter shape) injected by codegen around dispatch.
  • Grain versioning: tag interfaces with a version in the macro; directory refuses routing to older silos for newer versions during rolling deploys. This is the replacement for hot reload.
  • OpenTelemetry spans propagated through the call envelope; structured logging via stdout-capture or dedicated tracing API.

Out of scope

  • Distributed transactions (2PC across grains). Months of work, narrow use case. Recommend sagas with compensating actions instead. Deferred indefinitely.
  • Grain services / system targets. Niche Orleans features; revisit if needed.
  • Strongly-typed remote interface across language boundaries. Initial release is Rust-to-Rust grains; cross-language requires the wasmtime component model and stable IDL — possible on the modernised substrate but not in the first version.

Performance characteristics

Building Orleans after the substrate completion (vs on the current runtime) costs nothing on the steady-state per-call path — the new substrate primitives are additive and not on the hot path of send/receive. The completed path is modestly faster on:

  • Failover (monitors react in one hop vs poll-interval lag).
  • Reminder precision (host-side firing vs guest-side tick scan).
  • Fan-out (host-side group_send vs guest loop).

Versus native .NET Orleans, expect within ~10–20% on call throughput, with better tail latency (real preemption) and stronger fault isolation (WASM sandbox), at the cost of higher per-activation memory. The trade-offs land on the right side for typical Orleans workloads (gaming, IoT, durable services, business workflows), all of which are network- and persistence-bound.

Definition of done

The framework is "done" when a worked example exists where a multi-silo Lunatic cluster runs a virtual-actor service with persistent state, reminders, single-activation, automatic failover on node loss, and a typed Rust client. The framework crate has no unsafe outside FFI shims and depends only on rust-lib.

Each individual change inside any phase must also satisfy the per-change gates — tests, lint, and documentation in step with the code. See platform definition of done for the gates that apply to every increment.
